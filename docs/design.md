# Design Doc — слой интерпретации анкеты

## 1. Задача и принцип

Превратить сырые ответы анкеты `survey_answers` в структурированный `user_profile`,
который потребляет сборщик плана. Главный архитектурный принцип:

> **Вся логика — детерминированный код. LLM вызывается ровно один раз и только на
> одно поле со свободным текстом.**

Детерминированная часть полностью воспроизводима, тестируема и не зависит от сети.
LLM добавляет ровно одну вещь, которую нельзя сделать кодом: интерпретацию
произвольного человеческого текста — и при этом **не получает права** напрямую
писать поля профиля (см. §5).

## 2. Архитектура

```
                 config/*.yaml  (таблицы, маппинги, правила — ДАННЫЕ)
                        │ читается на старте, валидируется
                        ▼
 survey_answers ─► normalize ─► classify (goal / intensity / contra / flags / env)
   [JSON-вход]     [labels→codes]            │  всё детерминированно
                        │                    │
                        ▼                    │
                  free_text (str) ──► interpret  ←─ ОДИН вызов LLM (forced tool use)
                                          │            └─ или offline-stub
                                          ▼
                                      assemble  ──► user_profile
                              (мердж + сквозной флаг needs_human_review)
```

Слои (каждый — отдельный модуль в `src/`):

| Слой | Модуль | Тип | Ответственность |
|------|--------|-----|-----------------|
| Контракт | `contracts.py` | — | Схемы вход/выход (dataclasses + enum), сериализация в JSON |
| Конфиг | `config_loader.py` | код | Загрузка + валидация YAML; «упасть громко» на битом конфиге |
| Нормализация | `normalize.py` | код | `label → code`, сбор аномалий (неизвестные ответы/вопросы) |
| Классификация | `classify.py` | код | Цель, интенсивность, противопоказания, флаги, уровень оборудования |
| Свободный текст | `free_text.py` | **LLM** | Один вызов; интерпретатор + offline-stub за общим интерфейсом |
| Оркестрация | `pipeline.py` | код | Связывает слои, считает сквозной `needs_human_review`, собирает выход |

## 3. Поток данных (провенанс каждого поля)

Откуда берётся каждое поле выхода и через какой конфиг оно проходит:

| Поле `user_profile` | Источник (вопрос) | Через какой конфиг | Кто считает |
|---|---|---|---|
| `goal.{code,focus}` | Главная цель | `questions.yaml`, `goals.yaml` | `classify_goal` |
| `intensity.{level,score,drivers}` | Опыт + Частота + Минуты | `questions.yaml`, `intensity.yaml` | `classify_intensity` |
| `environment.locations` | Где занимаешься | `questions.yaml` | `normalize` |
| `environment.equipment_level` | Где занимаешься | `flags.yaml` (`equipment_level`) | `derive_equipment_level` |
| `constraints.limitations` | Ограничения / травмы | `questions.yaml` | `derive_contraindications` |
| `constraints.contraindicated_exercise_types` | Ограничения / травмы | `contraindications.yaml` | `derive_contraindications` |
| `flags.*` | разные поля | `flags.yaml` | `evaluate_flags` |
| `flags.needs_human_review` | флаги + сигнал LLM | — (код) | `pipeline._needs_human_review` |
| `free_text_signal.*` | Свободное поле | — (промпт+схема) | `interpret` (LLM/stub) |
| `meta.{unmatched_questions,warnings}` | весь вход | — | накапливается по пути |

## 4. Граница «конфиг vs код» — и почему так

Правило разделения, которым я руководствовался:

> **Конфиг = значения и таблицы, которые меняются вместе с анкетой или бизнес-правилами.
> Код = алгоритм, который их применяет.**

| В конфиге (`config/*.yaml`) | В коде (`src/*.py`) |
|---|---|
| Список вопросов, их типы и опции (`label↔code`) | Процедура нормализации |
| Цель → focus | Алгоритм выбора цели + дефолт |
| Очки за опыт/частоту/минуты + пороги бэндов | Суммирование очков и выбор бэнда |
| Травма → список тегов упражнений + словарь тегов | Объединение тегов, разбор «Нет»+травма |
| Декларативные правила флагов (`any_of/all_of/none_of/equals`) | Универсальный движок правил `_match` |
| Правила уровня оборудования (ordered, first-match) | Тот же движок |

**Почему именно так:** продуктовая команда меняет анкету постоянно — добавляет цель,
переименовывает опцию, правит, какие травмы что исключают. Всё это — изменения
**данных**, и они не должны требовать правки процедур и регрессии логики. А вот
«как считается интенсивность» или «как объединяются противопоказания» — это
**алгоритм**: он стабилен, его тестируем, и держать его в YAML было бы и опасно
(нет типов/тестов), и нечитаемо.

Важное следствие дизайна интенсивности: очки заданы так, что новичок (опыт = 0
очков) при любой частоте и длительности набирает максимум `0+2+2 = 4` → «moderate».
То есть **новичка нельзя автоматически отнести к «high»** — это свойство
безопасности, которое получается из конфига, а не из хардкод-исключения в коде.

## 5. LLM: ровно один вызов и ограниченные полномочия

* **Вход:** только строка из «Свободного поля». Если строка пустая — вызов не
  делается вообще (`source = "empty"`).
* **Выход:** маленькая закрытая схема `FreeTextSignal` — `summary`, `likes`,
  `dislikes`, `schedule_notes`, `mentioned_constraints[]`,
  `mentions_potential_contraindication`.
* **Как добиваемся структуры:** forced tool use (`tool_choice = {type: tool, …}`) —
  модель обязана вернуть JSON по нашей `input_schema`. Это самый надёжный способ
  получить структурированный вывод сегодня. Промпт требует «извлекай только то, что
  явно написано; не диагностируй; не советуй».
* **Ограниченные полномочия (ключевое решение):** модель **не пишет**
  противопоказания и поля профиля напрямую. Максимум, что она может по части
  безопасности, — **поднять флаг** `mentions_potential_contraindication`.
  Детерминированный код (`_needs_human_review`) превращает этот advisory-флаг в
  действие: ставит `needs_human_review = true`. Так пользовательский текст
  «недавно была операция на плече» гарантированно эскалируется, но
  непредсказуемый LLM никогда сам не меняет медицински значимый список
  `contraindicated_exercise_types`.
* **Деградация:** при пустом ответе модели, невалидном JSON или сетевой ошибке
  возвращается безопасный дефолт с `source = "error"` и `needs_human_review = true`
  (fail-safe, а не fail-open). Пайплайн не падает.
* **Тестируемость:** интерпретатор спрятан за интерфейсом `FreeTextInterpreter`.
  В тестах и при отсутствии `ANTHROPIC_API_KEY` подставляется детерминированный
  `StubFreeTextInterpreter` (keyword-матчинг), поэтому сквозной поток, включая
  ветку «флаг LLM → needs_human_review», воспроизводим оффлайн.

## 6. Обработка ошибок и наблюдаемость

Слой **никогда не падает** на «грязном» входе — он деградирует и пишет в
`meta.warnings`: неизвестный вопрос → `meta.unmatched_questions`; неизвестная опция
→ дропается с предупреждением; single-вопрос с >1 ответом → берётся первый;
отсутствующий обязательный ответ → консервативный дефолт + предупреждение.
Сборщик плана видит и профиль, и журнал предупреждений и может решать сам.

## 7. Расширяемость (ответ на бонус)

**Добавить новый вопрос с минимальной правкой кода.** Зависит от типа:

* **Если это просто новая опция/цель/травма у существующего вопроса** —
  правится **только конфиг**: строка в `questions.yaml` (label↔code) и, если нужно,
  строка в `goals.yaml` / `contraindications.yaml` / `flags.yaml`. Кода — ноль.
* **Если это новый вопрос, влияющий только через флаги** (например «Цель по
  времени: к дате / без срока») — добавляем вопрос в `questions.yaml` и правило в
  `flags.yaml`. Универсальный движок правил подхватит его без правок процедур.
* **Если новый вопрос вводит новое поле выхода с новой логикой агрегации**
  (например «оборудование дома: резина/турник/…» со своей сводкой) — тогда, и
  только тогда, добавляется: поле в `contracts.py`, маленькая функция в
  `classify.py` и её вызов в `pipeline.py`. Это честная граница: новые **данные**
  бесплатны, новый **тип вычисления** требует немного кода.

Та же модель масштабируется на версионирование контракта: `schema_version` уже в
выходе, так что downstream может мигрировать постепенно.

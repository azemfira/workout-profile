# Краевые случаи и допущения

Неоднозначности, найденные в задании, и принятые по ним решения. Принцип по
умолчанию: **слой не падает, деградирует консервативно и всё логирует в
`meta.warnings`** — downstream сам решает, что делать.

## Вход и сопоставление

1. **Вход приходит с текстом вопроса как ключом.** Сопоставление по тексту хрупко
   (пробелы, регистр, тире `–` vs `-`). Решение: матчим по
   **нормализованному** тексту (схлопывание пробелов + lower-case) через
   `questions.yaml`. *Допущение/рекомендация:* в проде анкета должна слать
   **стабильный `question_id`**, а не отображаемый текст — тогда переименование
   вопроса не ломает пайплайн.

2. **Неизвестный вопрос** (нет в `questions.yaml`) → не роняем; кладём в
   `meta.unmatched_questions` + предупреждение.

3. **Неизвестная опция** (label не из словаря вопроса) → дропаем эту опцию с
   предупреждением, остальные ответы обрабатываем.

4. **Дубликат блока одного вопроса** → берём первый, остальные игнорируем с
   предупреждением.

## Типы вопросов

5. **`single`-вопрос с несколькими ответами** → берём первый код + предупреждение.
   (Не падаем: лучше отдать профиль, чем 500.)

6. **Отсутствует обязательный одиночный ответ:**
   * нет цели → дефолт `maintain` (из `goals.yaml`) + предупреждение;
   * нет драйвера интенсивности (опыт/частота/минуты) → отсутствующий драйвер
     даёт **0 очков** (консервативно, тянет к «low») + предупреждение.

7. **Пустой массив `answers`** у multi-вопроса → пустой список кодов (валидно:
   человек ничего не выбрал).

## Противопоказания / травмы

8. **«Нет» вместе с реальной травмой** (противоречивый ввод) → реальные травмы
   приоритетны, сентинел `none` отбрасывается + предупреждение.

9. **Пересечение тегов у разных травм** (напр. `high_impact` и у спины, и у
   колен) → объединение без дублей; порядок тегов детерминированный
   (отсортирован) для воспроизводимого вывода.

10. **Маппинг травма→упражнения — упрощённый и НЕ медицинский совет.** Это
    разумный консервативный дефолт для тестового. *Допущение:* в проде таблицу
    `contraindications.yaml` должен ревьюить специалист; при `heart_pressure` /
    `pregnancy` мы дополнительно ставим `medical_clearance_recommended` и
    `needs_human_review`, не полагаясь только на список тегов.

## Интенсивность

11. **Что считать «интенсивностью» — не задано.** Решение: взвешенная сумма очков
    по опыту + частоте + длительности с порогами-бэндами (всё в `intensity.yaml`).
    Обоснование и свойство «новичок никогда не high» — см. design.md §4.

## Свободный текст и LLM

12. **Пустое свободное поле** → LLM **не вызывается** (`source = "empty"`),
    `provided = false`. Экономим вызов и не галлюцинируем на пустоте.

13. **Свободный текст вскрывает травму, которой нет в чекбоксах** (напр. «болит
    запястье», «операция на колене»). Решение: LLM **не добавляет**
    противопоказания сам (это недетерминированно и рискованно для безопасности).
    Он лишь поднимает `mentions_potential_contraindication`, а
    детерминированный код ставит `needs_human_review = true`. Так downstream/
    тренер обязательно посмотрит, но автогенерация плана не опирается на
    нестабильный вывод модели. *Допущение:* окончательную сверку травм из текста
    делает человек или отдельный, явно спроектированный шаг.

14. **LLM вернул мусор / упал / нет ключа.** Fail-safe: `source = "error"`,
    `needs_human_review = true`, пайплайн не падает. (Не fail-open: на сбое мы
    становимся осторожнее, а не наоборот.)

15. **Текст на любом языке.** Промпт явно допускает любой язык; примеры в репо —
    на русском.

## Контракт

16. **Эволюция контракта.** В выходе есть `schema_version` (сейчас `"1.0"`), чтобы
    downstream мог мигрировать постепенно при изменении схемы.

17. **`environment` как отдельный блок.** В задании среди обязательных полей он не
    назван явно, но «структурные флаги» и сборка плана без него неполны
    (нужно знать оборудование). Добавил `environment.{locations, equipment_level}`
    как полезное, но не противоречащее контракту расширение.

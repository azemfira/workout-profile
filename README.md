# Слой интерпретации анкеты → `user_profile`

Превращает сырые ответы онбординг-анкеты в структурированный `user_profile` для
сборщика плана тренировок.

**Принцип:** вся логика — детерминированный код; LLM вызывается **ровно один раз**
и только на поле со свободным текстом. Детерминированная часть на 100%
воспроизводима и покрыта тестами; анкету можно менять, правя **конфиг**, а не код.

## 1. Что реализовано

* Полный пайплайн `survey_answers → user_profile`: нормализация ответов,
  классификация цели/интенсивности/противопоказаний/флагов/оборудования,
  интерпретация свободного текста, сборка итогового профиля.
* Конфигурируемая анкета — вопросы, опции, цели, пороги интенсивности,
  противопоказания и правила флагов лежат в `config/*.yaml`, а не в коде.
* Свободный текст обрабатывается одним вызовом LLM (`anthropic`, forced tool
  use) либо offline-stub'ом (keyword-матчинг) — пайплайн работает без сети и
  без API-ключа.
* Безопасные деградации на «грязном» входе: неизвестные вопросы/опции,
  дубликаты, противоречивые ответы, пустой текст, сбой LLM — пайплайн не
  падает, а пишет предупреждения в `meta.warnings` и при необходимости ставит
  `needs_human_review = true`.
* 31 unit-тест (`tests/`) на нормализацию, классификацию, сборку и краевые
  случаи + сквозной прогон пайплайна со stub-интерпретатором.
* Документация: [`docs/design.md`](docs/design.md) — архитектура и обоснование
  решений; [`docs/edge_cases.md`](docs/edge_cases.md) — список краевых случаев
  и принятых по ним допущений.

### Структура

```
config/                  # ДАННЫЕ: меняем анкету здесь, без правки кода
  questions.yaml         #   единственный источник правды по вопросам/опциям/кодам
  goals.yaml             #   цель → focus
  intensity.yaml         #   очки + пороги бэндов интенсивности
  contraindications.yaml #   травма → теги упражнений (+ словарь тегов)
  flags.yaml             #   декларативные правила флагов + уровень оборудования
src/                     # КОД: процедуры/алгоритмы
  contracts.py           #   схемы вход/выход (dataclasses + enum)
  config_loader.py       #   загрузка + валидация конфига
  normalize.py           #   labels → codes
  classify.py            #   цель, интенсивность, противопоказания, флаги, оборудование
  free_text.py           #   ОДИН вызов LLM (forced tool use) + offline-stub
  pipeline.py            #   оркестрация: survey_answers → user_profile
tests/                   # unittest, без внешних зависимостей
examples/                # sample_input.json + run_example.py
```

## 2. Как запустить / проверить

Нужен Python 3.10+.

```bash
pip install -r requirements.txt
```

Для прогона тестов и оффлайн-демо достаточно `PyYAML`. Пакет `anthropic` нужен
только для **живого** LLM-вызова.

### Демо

```bash
python examples/run_example.py
# или со своим файлом:
python examples/run_example.py path/to/survey.json
```

Без `ANTHROPIC_API_KEY` свободный текст обрабатывает оффлайн-stub
(keyword-матчинг), поэтому команда всегда отрабатывает и печатает готовый
`user_profile` в JSON.

### Тесты

```bash
python -m unittest discover -s tests -v
```

31 тест на детерминированную часть (нормализация, классификация, сборка,
краевые случаи) + сквозной прогон пайплайна со stub-интерпретатором. Внешних
зависимостей и сети не требуется.

### Живой LLM-вызов (опционально)

```bash
pip install anthropic
export ANTHROPIC_API_KEY=sk-...
python examples/run_example.py
```

Тогда `examples/run_example.py` использует `LLMFreeTextInterpreter`: один вызов
Messages API с **forced tool use**, который заставляет модель вернуть JSON по
нашей схеме. Модель по умолчанию — `claude-haiku-4-5-20251001` (маленькая задача;
переопределяется переменной `WORKOUT_LLM_MODEL`). При сбое модели/сети пайплайн
не падает — отдаёт безопасный дефолт с `needs_human_review = true`.

### Использование как библиотеки

```python
from src import build_profile

profile = build_profile(survey_answers)   # интерпретатор выбирается автоматически
data = profile.to_dict()                   # dict, готовый к json.dumps
```

Интерпретатор можно внедрить явно (например, в тестах):

```python
from src.pipeline import build_profile
from src.free_text import StubFreeTextInterpreter

profile = build_profile(survey_answers, interpreter=StubFreeTextInterpreter())
```

## 3. Основные решения

* **Граница «конфиг vs код»**: всё, что меняется вместе с анкетой/бизнес-
  правилами (вопросы, опции, пороги, маппинги травм) — в `config/*.yaml`;
  алгоритм, который это применяет — в `src/`. Это позволяет менять анкету без
  регрессии логики. Подробнее и обоснование — [`docs/design.md`, §4](docs/design.md#4-граница-конфиг-vs-код--и-почему-так).
* **LLM вызывается ровно один раз и с ограниченными полномочиями**: только на
  свободном текстовом поле, через forced tool use (закрытая схема
  `FreeTextSignal`). Модель **не может** напрямую писать противопоказания или
  другие поля профиля — максимум, что она делает по части безопасности, это
  поднимает advisory-флаг `mentions_potential_contraindication`, который
  детерминированный код превращает в `needs_human_review = true`. Решение —
  чтобы непредсказуемый вывод LLM никогда не определял медицински значимые
  поля напрямую. См. [`docs/design.md`, §5](docs/design.md#5-llm-ровно-один-вызов-и-ограниченные-полномочия).
* **Fail-safe, а не fail-open**: любой сбой (сеть, невалидный JSON, неизвестный
  вопрос/опция, противоречивый ввод) деградирует консервативно — пайплайн не
  падает, но и не «угадывает» в опасную сторону; вместо этого пишет
  предупреждение в `meta.warnings` и/или поднимает `needs_human_review`.
* **Очки интенсивности подобраны так, что новичок никогда не попадает в
  «high»** (максимум `0+2+2=4` → «moderate») — это свойство безопасности,
  заложенное в конфиг, а не хардкод-исключение в коде.
* **`schema_version` в выходе** — чтобы downstream-потребитель мог мигрировать
  постепенно при изменении контракта.
* **`environment` как отдельный блок** — в задании не назван явно среди
  обязательных полей, но без информации об оборудовании сборка плана неполна;
  добавлен как расширение, не противоречащее контракту.

Полный список краевых случаев и принятых по ним решений — в
[`docs/edge_cases.md`](docs/edge_cases.md).

## 4. Допущения / что не успели

* **Сопоставление вопросов идёт по нормализованному тексту**, а не по
  стабильному `question_id`. Для прода правильнее, чтобы анкета слала
  `question_id` — переименование текста вопроса тогда не ломает пайплайн.
* **Маппинг «травма → противопоказанные упражнения» (`contraindications.yaml`)
  упрощённый и не является медицинским советом** — это консервативный дефолт
  для тестового задания. В проде таблицу должен ревьюить специалист.
* **Сверка травм, упомянутых в свободном тексте, но не отмеченных в
  чекбоксах, не автоматизирована** — LLM только поднимает флаг
  `mentions_potential_contraindication` и эскалирует на `needs_human_review`;
  финальное решение оставлено человеку или отдельному, явно спроектированному
  шагу.
* Не реализована персистентность/API-слой (хранение профилей, HTTP-эндпоинт)
  — задача ограничена чистой трансформацией `survey_answers → user_profile`
  как библиотекой; интеграция с сборщиком плана тренировок не входила в
  объём работы.
* Промпт для LLM допускает текст на любом языке, но **тестировался только на
  русском** — поведение на других языках не покрыто тестами.

## Контракт (кратко)

**Вход** `survey_answers`: `[{ "question": str, "answers": [str, ...] }, ...]`.

**Выход** `user_profile`:

```jsonc
{
  "schema_version": "1.0",
  "goal":        { "code": "build_muscle", "focus": "hypertrophy" },
  "intensity":   { "level": "moderate", "score": 3,
                   "drivers": { "experience": "...", "frequency": "...", "session_minutes": "..." } },
  "environment": { "locations": ["gym", "home_dumbbells"], "equipment_level": "full" },
  "constraints": { "limitations": ["knees"],
                   "contraindicated_exercise_types": ["deep_knee_flexion", "high_impact", "plyometric"] },
  "flags":       { "has_gym_access": true, "...": false, "needs_human_review": true },
  "free_text_signal": { "provided": true, "source": "llm|stub|empty|error", "summary": "...",
                        "preferences": { "likes": [], "dislikes": ["бег"] },
                        "schedule_notes": null, "mentioned_constraints": [],
                        "mentions_potential_contraindication": false, "raw_text": "..." },
  "meta":        { "unmatched_questions": [], "warnings": [] }
}
```

Точные типы и допустимые значения — в `src/contracts.py`.
